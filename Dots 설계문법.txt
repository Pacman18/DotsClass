========================== Dots EntityCommandBuffer 사용 ===============================

EntityCommandBuffer(ECB)의 역할
EntityCommandBuffer는 EntityManager에 직접적으로 명령을 내리지 않고,
여러 엔티티 관련 명령(생성, 삭제, 컴포넌트 추가/제거 등)을 버퍼에 임시로 저장해두는 역할을 합니다.
이렇게 하는 이유는,
ECS의 Job이나 시스템 내부에서 직접적으로 EntityManager를 수정하는 것이 위험하거나 불가능한 경우가 많기 때문입니다.
(특히 병렬 처리나, 쿼리 중간에 엔티티를 삭제/생성하는 경우)
2. ecb.DestroyEntity(entity)의 역할
이 코드는 "이 엔티티를 삭제해달라"는 명령을 ECB에 저장합니다.
실제로 이 시점에서는 엔티티가 삭제되지 않습니다.
3. ecb.Playback(state.EntityManager)의 역할
Playback을 호출하면,
ECB에 저장된 모든 명령(여기서는 DestroyEntity 등)을 실제로 EntityManager에 적용합니다.
즉, 이 시점에 버퍼에 쌓인 엔티티 삭제 명령이 실제로 실행되어
해당 엔티티가 월드(World)에서 삭제됩니다.
4. 전체 흐름 요약
ECB 생성
→ 명령을 임시로 저장할 버퍼 준비
명령 저장
→ ecb.DestroyEntity(entity) 등으로 삭제할 엔티티를 버퍼에 저장
Playback 호출
→ 버퍼에 저장된 명령을 실제 EntityManager에 적용(실제 삭제 등)
그림으로 표현
Apply to SantaNetcode...
결론
ECB: 엔티티 관련 명령을 임시로 저장하는 버퍼
DestroyEntity: 삭제 명령을 버퍼에 저장
Playback: 버퍼에 저장된 명령을 실제로 실행(EntityManager에 적용, 월드에서 삭제 등)
추가로 궁금한 점 있으면 언제든 질문해주세요!

ComponentTypeHandle<T>는 특정 컴포넌트 타입(T)에 접근할 수 있도록 도와주는 **핸들(Handle)**입니다. 이 핸들을 통해 Job 내부에서 컴포넌트 데이터를 안전하고 빠르게 읽거나 쓸 수 있습니다.


** 중요 포인트 ArchetypeChunk는 같은 Archetype (즉, 같은 컴포넌트 조합)을 가진 엔티티들을 담고 있습니다 ** 

m_Query = GetEntityQuery( ComponentType.ReadWrite<Position>(),ComponentType.ReadOnly<Velocity>());
state.Dependency = job.ScheduleParallel(m_Query, Dependency); <=========== Archetype의 청크를 미리 쿼리로 뽑아서 넣어주니까 문제없다.

public void Execute(ArchetypeChunk chunk, int chunkIndex, int firstEntityIndex)
{
	// 쿼리로 뽑아서 넣어준거라 둘의 배열 숫자가 같을 수 밖에 없다. 
    var positions = chunk.GetNativeArray(positionTypeHandle) or (ref positionTypeHandle) ; // ref 형식으로 사용하네 
    var velocities = chunk.GetNativeArray(velocityTypeHandle);

    for (int i = 0; i < chunk.Count; i++)
    {
        positions[i] += velocities[i] * deltaTime;
    }
}

1. WithAll<T1, T2, ...>
설명:
지정한 모든 컴포넌트가 붙어있는 엔티티만 쿼리합니다.
예시:
Apply to InputSystem....
→ T1과 T2가 모두 붙어있는 엔티티만 선택
적용:
A: T1, T2, T3 (O)
B: T1, T2 (O)
C: T3 (X)
결과: A, B
2. WithAny<T1, T2, ...>
설명:
지정한 컴포넌트 중 하나라도 붙어있는 엔티티를 쿼리합니다.
예시:
Apply to InputSystem....
→ T1이 붙어있는 엔티티 모두 선택
적용:
A: T1, T2, T3 (O)
B: T1, T2 (O)
C: T3 (X)
결과: A, B
Apply to InputSystem....
→ T2 또는 T3가 붙어있는 엔티티 모두 선택
A: T1, T2, T3 (O)
B: T1, T2 (O)
C: T3 (O)
결과: A, B, C
3. WithNone<T1, T2, ...>
설명:
지정한 컴포넌트가 하나도 붙어있지 않은 엔티티만 쿼리합니다.
예시:
Apply to InputSystem....
→ T1이 없는 엔티티만 선택
적용:
A: T1, T2, T3 (X)
B: T1, T2 (X)
C: T3 (O)
결과: C

// 쿼리에서 가져올 수 있다. 
void PrintColliderInfo(PhysicsCollider physicsCollider)
{
    var colliderRef = physicsCollider.Value;
    if (!colliderRef.IsCreated)
    {
        UnityEngine.Debug.Log("Collider가 없음!");
        return;
    }

    // Collider 타입 확인
    var colliderType = colliderRef.Value.Type;

    switch (colliderType)
    {
        case ColliderType.Box:
            var boxCollider = (BoxCollider*)colliderRef.GetUnsafePtr();
            var box = boxCollider->Geometry;
            UnityEngine.Debug.Log($"Box: Center={box.Center}, Size={box.Size}, Orientation={box.Orientation}");
            break;

        case ColliderType.Sphere:
            var sphereCollider = (SphereCollider*)colliderRef.GetUnsafePtr();
            var sphere = sphereCollider->Geometry;
            UnityEngine.Debug.Log($"Sphere: Center={sphere.Center}, Radius={sphere.Radius}");
            break;

        case ColliderType.Capsule:
            var capsuleCollider = (CapsuleCollider*)colliderRef.GetUnsafePtr();
            var capsule = capsuleCollider->Geometry;
            UnityEngine.Debug.Log($"Capsule: Vertex0={capsule.Vertex0}, Vertex1={capsule.Vertex1}, Radius={capsule.Radius}");
            break;

        // 기타 타입도 필요하면 추가
        default:
            UnityEngine.Debug.Log($"ColliderType: {colliderType}");
            break;
    }
}

이렇게 선언하면,
이 구조체는 RPC 데이터로 사용될 수 있습니다.
실제로 RPC를 보내려면
IRpcCommand를 상속한 컴포넌트를 엔티티에 붙이고
SendRpcCommandRequest 컴포넌트도 같이 붙여야
→ NetCode가 이 엔티티를 "RPC 메시지"로 인식해서 전송합니다.
3. SendRpcCommandRequest가 없으면?
없으면:
해당 엔티티는 RPC로 전송되지 않습니다.
즉,
IRpcCommand만 붙어있고 SendRpcCommandRequest가 없으면
→ NetCode는 이 엔티티를 "일반 데이터"로만 취급하고,
네트워크로 전송하지 않습니다.
4. 요약 (상관관계)
IRpcCommand:
"이 구조체는 RPC 데이터다"를 표시 (데이터 정의)
SendRpcCommandRequest:
"이 엔티티를 RPC로 전송해라"를 표시 (전송 트리거)
둘 다 필요
→ 엔티티에 두 컴포넌트가 모두 붙어 있어야
→ NetCode가 해당 엔티티를 RPC로 전송

IRpcCommand는 데이터 정의, SendRpcCommandRequest는 전송 트리거
둘 다 있어야 정상적으로 RPC가 동작
